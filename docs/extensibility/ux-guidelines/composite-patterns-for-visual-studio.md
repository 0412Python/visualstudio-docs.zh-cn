##  <a name="BKMK_DataVisualization"></a>数据可视化  
  
### <a name="overview"></a>概述  
 图是一种可视方式进行聚合和可视化数据处理以增强决策。 它们可以帮助用户面临着大量的数据，但很少，则表示看到什么都应该注意，而且新增功能可能需要一项操作。  
  
 如果任意以下条件为 true，用户将从图表中受益：  
  
-   图表将帮助用户识别它们可以会对操作的任务？  
  
-   图表将使用户能够预测的潜在的更改的后果？  
  
-   图表将帮助用户发现趋势并标识模式？  
  
-   图表将使用户可以更好的决策？  
  
-   将图表帮助回答用户可能在给定的上下文中具有特定问题？  
  
#### <a name="general-rules-for-charts"></a>用于图表的一般规则  
  
-   清楚地标签数据。 图例而无需说明只是美观的图片。  
  
-   启动从零开始，以避免倾斜比例的轴。 行长度和栏大小是重要的视觉提示，了解数据点之间的关系。  
  
-   创建图表，不信息图。 信息图是艺术表示的数据，以及其主要目标是 visual 讲故事。 图表可以 （并且应该） 是好的视觉效果，但让数据自己说话。  
  
-   避免 skeumorphism、 图形化的条形图、 对比度 hashmarks 和其他信息图收尾工作。  
  
-   不要将三维效果用作装饰性元素。 仅当使用它们它们真正必不可少的用户能够理解的信息。  
  
-   避免使用多个线条和填充，因为两个以上的颜色使这种图表类型难以阅读和正确解释。  
  
-   不使用图表 （或者任何图） 作为了解概念或与数据交互的唯一方法。 这会带来有视觉障碍的用户的问题。  
  
-   不要将用作免费或装饰性元素在页面的图表。 换而言之，如果图表不会添加任何值或帮助用户解决问题，不要使用它。  
  
### <a name="chart-types"></a>图表类型  
 在 Visual Studio 中使用的图表的类型包括条形图、 折线图、 已修改的饼图称为环图表或"环形图，"时间线，散点图 （也称为"群集图表"） 和甘特图。 每种类型的图表可用于通信不同类型的信息。  
  
### <a name="other-charting-considerations"></a>其他图表制作的注意事项  
  
#### <a name="color"></a>颜色

##  <a name="BKMK_OnObjectUI"></a>对象上的 UI 和扫视  
 本部分向上下文扫视，也称为代码查看视图中，一种类型的唯一到 Visual Studio 的对象上的 UI。  
  
### <a name="overview"></a>概述  
  
-   对象上的 UI 应授予用户详细信息或交互性而不使其主要任务。  
  
-   Visual Studio 中的对象上的 UI 的主模式被称为"前提关注的信息。"  
  
-   Visual Studio 中的对象上的 UI 为内联、 浮点和是持久或暂时。  
  
    -   代码查看视图中，一种类型的对象上的 UI 在 Visual Studio 中，是内联和持久。  
  
    -   CodeLens，一种类型的对象上的 UI 在 Visual Studio 中，为浮点型和暂时性  
  
 了解如何工作的一段代码，或查找有关该代码的详细信息通常要求开发人员可以切换上下文并转到其他内容或另一个窗口。 这些上下文变化可能中断性的因为如果他们离开其主窗口，用户可能会失去其初始任务的焦点。 此外，获取原始上下文后，可能很困难，尤其当切换 windows 导致被遮盖的其他 UI，其原始代码。  
  
 对象上的 UI 遵循模式称为"前提关注的信息。" 这些消息、 弹出窗口和对话框为用户提供添加说明或交互性，而不会丢失专注于其主要任务的其他相关信息。 对象上的用户界面的示例包括当用户悬停在通知区域中的图标、 拼错的字和 Visual Studio 2013 中引入的扫视视图下的红色波形曲线上其指针时，显示的弹出窗口。  
  
### <a name="decision-points"></a>决策点  
 在 Visual Studio 中，有多种方式来使用此模式在关注的信息。 选择正确的机制和实现一致的、 可预测的方式是至关重要的总体体验。 否则，用户可能会显示出会降低焦点从内容本身的令人困惑或不一致体验。  
  
#### <a name="relationships-between-master-and-detail-content"></a>Master 和内容的详细信息之间的关系  
 在关注的信息用于显示关系内容之间的用户是关注 （"主"内容） 和其他相关内容 （"从"内容）。 在此模式中，详细信息内容都清楚地与用户正在使用，并可以显示接近母版页的内容的内容。 补充信息或不能进行汇总而无需绝大多数母版页的内容的信息应遵循另一种模式，如工具窗口。  
  
-   **始终**显示中紧靠母版页的内容的详细信息内容。  
  
-   **始终**确保详细内容仍使用户能够继续专注于母版页的内容。 通常，实现此目的的最佳方式是呈现作为接近作为可能的主内容的详细信息内容。 可以呈现旁边母版的内容，一个弹出窗口中的详细信息内容或呈现下方母版页的内容的详细信息内容内联执行此操作。  
  
-   **永远不会**使用在离开主内容用户的关注点的信息。 如果用户需要单独查看详细信息内容，公开使用户能够执行此操作执行显式操作。  
  
#### <a name="design-details"></a>设计的详细信息  
 在你确定了对象上的用户界面是正确的选择，有四个主要设计注意事项：  
  
1.  **持久性：**内容应为持久的还是暂时性？   
    用户想要的信息保持可见引用或与之交互？ 或者，将用户想要快速速览的信息，然后继续其主要任务？  
  
2.  **内容类型：**将的内容是信息性、 可操作，或导航？   
    用户是否需要有关 master 内容的更多详细信息？ 用户是否需要完成一个任务，它会影响主内容？ 或者，用户需要要定向到另一个资源？  
  
3.  **指示器类型：**环境指示器是否有意义？   
    可以信息汇总以有用的方式和显示没有超载母版页的内容？  
  
4.  **笔势：**什么笔势将用于调用和关闭用户界面？   
    如何将用户显示的详细信息内容并将其发送离开？ 在添加如固定暂时性的而且持久状态之间进行切换的笔势是否有值？  
  
 在对象上的用户界面的主要组件，每个这些四个决策点会产生影响。  
  
### <a name="on-object-ui-components"></a>对象上的 UI 组件  
  
1.  容器 （内容显示器） 类型  
  
    -   浮动  
  
    -   内联  
  
2.  内容类型  
  
    -   可能是静态或动态的信息： 数据  
  
    -   更改主内容的可操作： 命令  
  
    -   将用户带到另一个窗口或应用程序，如 MSDN 的导航： 链接  
  
3.  笔势  
  
    -   调用  
  
    -   开除  
  
    -   固定  
  
    -   其他交互  
  
4.  持久性和提交模型  
  
    -   暂时性故障  
  
    -   Durable  
  
    -   自动  
  
    -   按需  
  
5.  （可选） 的环境指示器  
  
    -   波形曲线下划线  
  
    -   智能标记图标  
  
    -   环境的其他指标  
  
#### <a name="container-content-presenter-type"></a>容器 （内容显示器） 类型  
 有两个主要选项可用于呈现在关注点的内容：  
  
1.  **内联：**内联演示程序，如扫视视图引入在 Visual Studio 2013 代码编辑器中，通过转换现有内容，使新内容的空间。  
  
    -   **首选**你呈现的内联演示如果你希望用户会想要花费很长时间中引用或内容与之进行交互。  
  
    -   **避免**内联演示如果你希望用户将需要粗略地看一下呈现，然后继续其主要任务与很少发生中断的信息。  
  
2.  **浮动：**浮点演示器位于接近尽可能所选内容，但不会更改现有内容的布局。 可以采用不同的策略，例如通过显示浮动内容面板最接近的可用到所选符号的空白区域。  
  
    -   **首选**浮点演示者，如果希望用户将需要粗略地看一下呈现，然后继续其主要任务与很少发生中断的信息。  
  
    -   **避免**浮点演示者，如果希望用户将需要花费大量时间引用或内容与之进行交互的存在你。  
  
#### <a name="content-type"></a>内容类型  
 有三种主要类型可以在任何对象上的 UI 容器内显示的内容。 可以显示这些类型的信息的任意组合。 三种类型是：  
  
1.  **信息：**大多数对象上的 UI 容器将显示一些种类的信息性内容。 内容可以表示环境的当前状态有关的信息也可能表示环境的潜在将来状态有关的信息。 例如，它无法用于显示特定的命令，如重构，对现有代码的效果。  
  
    -   **始终**使用的规范表示形式显示的信息。 例如，代码应如下所示代码中完成，但语法突出显示，并应遵守任何字体和其他用户已设置的环境设置。  
  
    -   **始终**考虑对信息性内容，如果该相同信息会显示为 master 的内容，就支持任何操作。 例如，如果提供的对象上的 UI 容器内的现有代码，务必考虑支持浏览和修改该代码的能力。  
  
    -   **始终**请考虑使用不同的背景色，如果提供信息性内容表示潜在的将来状态。  
  
2.  可操作： 某些对象上的 UI 容器将提供对主机的内容，如执行重构操作执行某些操作的能力。  
  
    -   **始终**定位分开条信息性内容可操作的命令。  
  
    -   **始终**启用和禁用操作在适当的时候。  
  
    -   **始终**指表示对话框中的命令的标准准则。  
  
    -   **始终**保持在绝对对象上的 UI 容器中公开的操作的数量最小。 与对象上的用户界面交互应为轻量、 快速的体验。 用户不必展开上管理对象上的 UI 容器本身的能量。  
  
    -   **始终**考虑如何以及何时将关闭或解除对象上的 UI 容器。 作为最佳做法，到结束之间的 master 和内容的详细信息对话框任何操作也应关闭对象上的 UI 容器，调用该操作时。  
  
3.  **导航：**某些对象上的 UI 容器包括能够将用户带到另一个窗口或应用程序，例如在用户的 web 浏览器中打开 MSDN 文章的链接。  
  
    -   **始终**前面预置"打开"与任何导航链接，以便用户将不会对感到惊讶通过要导航到一些其他内容。  
  
    -   **始终**导航链接分开可操作的链接。  
  
#### <a name="ambient-indicators-optional"></a>（可选） 的环境指示器  
 环境的指示器可以是细微，包括以对比颜色其余代码，从显示的文本或很明显，包括 tickler 符号，如波形曲线下划线和智能标记图标。 环境指示器通信相关的其他信息的可用性。 理想情况下，它们甚至无需用户与之交互提供有用信息。  
  
-   **始终**定位环境指示器，以便不会打乱或占用大量的用户。 如果无法在这种方式中定位的环境的指示器，请考虑其他解决方案。  
  
-   **始终**环境指示器与相关的内容尽可能近的位置。  
  
-   **始终**尝试创建一个指示符，总结了可提供的信息。 请考虑提供可用的数据项目数的计数 （例如，"3 参考"而不是只需"引用"），或者认为的某种其他方式将数据汇总。  
  
    -   在其中一个指示器的数据不能始终进行计算和显示的情况下，立即考虑提供渐进式反馈，因为其值的计算。 例如，考虑反映到可用的数据，类似于 Windows Phone 上的电子邮件实时磁贴刷新作为数目的增加而未读电子邮件的方式更新的更改进行动画处理。  
  
-   **永远不会**添加多个指示器不是用户可以合理地在中执行给定的一段的内容。 环境指示器应有用，而无需用户进行任何互动。 如果它们需要溢出和其他管理控件，可将它们放入视图，指示器将失去其环境。  
  
#### <a name="gestures"></a>笔势  
 允许用户继续关注于主内容的一个重要方面是通过支持正确的手势来打开和关闭更多详细信息内容。  
  
-   **始终**需要用户执行某些显式手势，若要打开的其他内容。 常见打开手势包括：  
  
    -   **悬停时：**工具提示或非交互式条信息性内容  
  
    -   **显式命令：**内联演示器  
  
    -   **双击环境指示器：** CodeLens 弹出窗口  
  
-   **始终**每当用户按 Esc 键关闭详细内容。  
  
-   **始终**考虑的对象上的 UI 上下文。 允许在容器内的交互的内容演示者，请仔细考虑是否显示其他信息的悬停时，即，可能需要对用户的工作流中断。  
  
-   **永远不会**似乎是可编辑或邀请用户交互的悬停时的显示内容。 此行为可以会让用户灰心如果他们尝试将光标悬停在详细信息内容，如工具提示的标准行为是以立即关闭光标位于时不再的主控形状上方生成它的内容。  
  
##  <a name="BKMK_SelectionModels"></a>所选内容模型  
  
### <a name="overview"></a>概述  
 所选内容模型是用于指示并确认对一个或多个对象中的用户界面相关的操作的机制。 本主题讨论选择交互模式中的 Visual Studio 文档编辑器： 文本编辑器、 设计图面和建模图面。  
  
 用户必须通过它们，正在向 Visual Studio 指示的方法和 Visual Studio 必须响应可预见的方式提供反馈的有关它在操作的用户。 差异或在用户和用户界面之间通讯问题可能会导致用户未注意到操作，这将产生意外的后果。 通常情况下，该错误不明显，直到用户将看到的内容丢失或已更改。 因此，所选内容模型是一个用户界面设计的最关键部分。 尽管 Visual Studio 中的所选内容模型是与 Windows 一致，而略有差异。  
  
 在 Visual Studio 中，如下所示 Windows 中，选择模型而异发生交互操作的上下文。 选择出现在四种类型的对象：  
  
-   Text  
  
-   图形对象  
  
-   列表和树  
  
-   网格  
  
 在这些对象，有三种类型的选择：  
  
-   连续  
  
-   不连续的  
  
-   Region

#### <a name="context"></a>上下文  
 Visual Studio 设计为保留的上下文中，强概念的用户正在跟踪。 只有一个窗口处于活动状态，，无论它是一个工具或文档窗口。 但是，最顶层的文档窗口始终保留潜在的选择。 尽管焦点可能是在工具窗口中，最后处于活动状态的文档窗口将显示所选内容，即使处于非活动状态。 这样做是为了保留它们已编辑的显示，以便他们可以返回和工具窗口与文档窗口之间无缝移动，Visual Studio 已保留其状态的文档中的用户的上下文。  
  
### <a name="text-selection"></a>文本选择  
 Visual Studio 编辑器中严格文本，如内置的文本编辑器中，使用相同的文本选择模型和中所述的外观[鼠标和指针](https://msdn.microsoft.com/en-us/library/dn742466.aspx)页的 Windows 上的用户体验交互准则MSDN。 在文本编辑器中的输入的焦点指示由竖线调用插入点。 插入点位于单像素粗线和彩色作为的任何内容将显示在它后面的反向属性。 根据设置的速率，它就闪烁**指针闪烁速率**中设置**速度**选项卡**键盘**控制面板中的小程序。  
  
#### <a name="contiguous-and-disjoint-selection"></a>连续和非连续选择  
 在文本编辑器中的选择仅是连续的。 不连续的文本选择不允许使用，但应在图形对象编辑器中进行寻址。 当用户的鼠标指针位于文本区域时，光标将更改为 i 型。 通过单击放置在文本编辑器中的单击位置中的插入点。 按住鼠标按钮开始选择突出显示和松开鼠标按钮结束选择突出显示。  
  
#### <a name="region-selection-box-selection"></a>区域选择 （框选择）

### <a name="graphical-selection"></a>图形选择  
  
#### <a name="interaction"></a>交互  
 图形对象选择可能十分复杂，而且取决于许多因素影响：  
  
-   **编辑器的主要选择模型。** 包含图形对象的编辑器还可以用于编辑文本或网格形式。 例如，编辑器中可能是一种基于文本的编辑器，还支持的图形对象，如 Visual Studio XAML 设计器的位置。 支持多个对象类型可能会影响用户如何选择的不同类型的对象组成的组。  
  
-   **支持的主要和次要选择状态。** 编辑器可以提供主要和次要选择状态，以便可以在协同中, 编辑对象相互对齐，调整大小在一起，依次类推。  
  
-   **就地编辑的支持。** 编辑器还可以允许要编辑其图形对象的内容。 例如，矩形形状可能还包含可以由用户更改内的文本。 此外，该文本可能居中或两端对齐。 就地编辑涉及用户交互的更详细的级别，因此需要一组适当的视觉提示，用于向用户显示状态信息。

|拖动|即使没有以前选择的对象，将指针更改为移动光标，将对象移动。|

#### <a name="graphical-containers"></a>图形容器  
 在某些情况下，图形编辑器的其他图形对象，如 Windows 窗体 Panel 控件或 HTML 设计器中的网格布局控件提供容器。 如果你的编辑器为其他图形对象提供了容器，然后应为容器仅 （上面所述的标准模型作为该容器项中的对象） 使用以下所选内容模型：

### <a name="view-selection-models"></a>查看所选内容模型  
  
#### <a name="tree-view"></a>树视图  
 与简单突出显示一起显示在树视图中的选择。 如果用户单击节点名称或节点图标上时，将成为选择的节点。 节点的左侧的三角标志符号展开或协定的树控件，但不是会影响用户的选择，有一个例外： 在折叠的父节点，该节点的子节点上所选内容时，所选内容将移动到父。  
  
 ![Visual Studio 中的典型树视图](../../extensibility/ux-guidelines/media/0713-25_treeview.png "0713年 25_TreeView")

 树视图可以支持连续和非连续选择，甚至跨树中的多个级别。 连续的或连续必须可见的树节点上进行多个选择。 如果一个节点处于折叠状态，不连续选择将丢失，并且已折叠节点获取所选内容。 在这种方式，用户可以看到将受影响的操作的节点。 当节点处于折叠状态时，它将变为不清楚的节点可能会受到影响。  
  
 选中的父节点时，可能有必要为适用于父级和子级的所有操作的情况下，但在应父级，应用该操作。 在这种情况下，在操作时，如复选框或确认对话框中对用户进行的"应用于所有子项"选项显式提供其他 UI。

|输入|结果|

#### <a name="selection-within-lists-and-grid-controls"></a>列表和网格控件中选定内容  
 在列表中选择的关键概念在于，它是基于行的表示该参数时进行选择整个行作为一个单元选择。 与此相反，网格可以允许特定的单元格要选择而不会影响的行的任何其他方面。 网格还可能包含嵌套行 （如树网格） 允许的层次结构，可以通过与父行交互来取消选择和选择整个分支层次的结构。 上的数据的整个行的简单突出显示颜色显示为列表中的选择。 单像素虚线边框的当前可编辑的行或单元格 （行如果所有单元格是只读的） 显示为焦点。  
  
> [!NOTE]
>  **焦点**和**选择**是不同的概念。 *焦点*表示的用户界面的元素为目标以接收输入未显式定向到另一个对象，而*选择*指状态的对象包含在一组对哪些后续对象的状态操作可能会发生。  
  
 列表中的选择可能是连续、 连续的或区域。 当多个选择是允许、 连续且不连续选择应始终支持，对区域 （框） 选择的支持时是可选的。 通过将列表正文空白区域中拖动启动区域选择。  
  
|对象|选择|  
|------------|---------------|  
|列表|连续|始终受支持 （如果允许多重选择）。|  
|列表|不连续的|始终受支持 （如果允许多重选择）。|  
|列表|Region|可选支持。 启动通过拖动鼠标列表正文的空白区域。|  
  
 在列表中上一次单击选择单击发生处的行。 如果用户碰巧支持就地编辑的列表单元内单击，该单元格也会立即激活就地编辑。 否则为整个行立即选择，并显示突出显示。  
  
 在列表正文中拖动会执行三个操作之一：  
  
-   如果列表支持它，并且鼠标按下的空白区域中，启动区域选择  
  
-   如果列表单元格或行支持正在拖动源，启动拖放操作  
  
-   选择当前行  
  
##### <a name="in-place-editing"></a>在就地编辑  
 允许就地编辑时有两种基本模式： 简单的编辑控件和属性选择器。 有一个简单的编辑控件的内容是突出显示并可供用户输入就会立即激活就地编辑。 属性选取器实现时，会显示属性选取器时，将调用该按钮，一旦激活就地编辑模式，并且未突出显示当前所选内容。 在单元格中应为右对齐选取器按钮。 有关就地编辑示例，请参阅**属性窗口**和**任务列表**Visual Studio 中。  
  
##### <a name="keyboard-support"></a>键盘支持  
 在列表和网格中选择键盘支持遵循标准的 Windows 约定：  
  
-   箭头键导航列表中，选择每个行/单元，因为移动焦点。  
  
-   Shift + 箭头的方向箭头键执行连续选择。  
  
-   Ctrl + 空格键切换跟有之间添加和删除列表项从选择创建非连续选择的箭头。  
  
-   包含嵌套层次结构的网格，向右箭头键将父行中，扩展和向左箭头键会将一个折叠。  
  
-   Tab 键将焦点移在当前行中单元格之间如果单元格都可编辑。  
  
-   Enter 键对列表中的项执行默认命令 (通常**打开**)。  
  
-   按下 F2 键激活就地编辑当前所选单元格。

### <a name="window-configurations"></a>窗口配置  
 窗口配置是基本演示开发环境的-它是一种方案组成的存在的工具窗口的列表以及排列它们的方式。 对于由 IDE （IDE 窗口） 的 windows，布局信息将保留每个用户，因此当 IDE，窗口布局显示与相同用户启动时持续退出 Visual Studio。 状态和位置的 IDE 窗口将保留在一个自定义选项 XML 格式文件中。 创建的包加载到 IDE 的工具窗口保留其注册表中的状态信息并可能或可能不是每个用户。  
  
#### <a name="profile-specific-layouts"></a>配置文件特定布局  
 每个配置文件包括工具窗口布局，方式为专用开发人员熟悉组织 (Visual c + + 开发人员希望看到**解决方案资源管理器**IDE，而 C# 开发人员希望看到左侧**解决方案资源管理器**右侧)。 用户选择在启动时的配置文件后加载配置文件特定窗口布局。 程序包作者应确定最适合其客户的体验，窗口布局知道随后将保存到窗口配置的用户进行的更改。  
  
##  <a name="BKMK_TouchInput"></a>触摸屏输入  
 用户日益在触摸设备上使用 Microsoft 开发产品。 但是，有一些难以在触摸设备上使用开发工具的障碍。 用户将希望我们的产品中提供可靠，并且是精确触摸体验。 这些准则的目的是通知以合并，并鼓励一致触摸体验跨 Visual Studio 和相关的产品哪些触摸功能有关的决策。  
  
### <a name="levels-of-experience"></a>级别的体验  
 以下级别的体验旨在作为指南来帮助团队确定哪些触摸功能提供基于其所需的投资兴趣触摸级别。  
  
-   **基本体验**是想要提供的团队 touch 功能，因此没有在其工作整个没有死结束。  
  
-   **优化体验**可供想要提供最常用的触摸功能 （例如，那些通常可以在 internet 浏览器应用程序） 的团队。  
  
-   **提升体验**是对于想要添加此类功能的团队一样笔势或其他可使其应用程序的可选功能触摸优先友好。  
  
||基本体验|优化的体验|提升的体验|  
|-|----------------------|--------------------------|-------------------------|  
|使用户可以...|修复代码和阅读并不死结束的解决方案/项目的级别|执行维护、 重构方法和导航任务|置信度在一致、 直观和流畅的体验操作|  
|编辑器|触摸平移和所选内容<br /><br /> 滚动条触摸跳转和按并拖动|捏合缩放<br /><br /> 快速滚动<br /><br /> 选择<br /><br /> 易于使用的上下文菜单||  
|顶部的工具窗口|列表平移<br /><br /> 项选择<br /><br /> 滚动条触摸跳转和按并拖动|轻松项滚动和所选内容||  
|窗口||重设窗口大小<br /><br /> 快速访问||  
|文档井中||打开的文件之间轻松导航||  
|笔势||确保在 IDE 的常见手势工作|基于笔势的操作<br /><br /> 支持拖放和设计器|  
|其他注意事项|||自定义屏幕键盘|
